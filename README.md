# Bincom Election Explorer

This Django solution answers the Bincom assessment using the provided `bincom_test.sql` dump as the source of truth.

## What it covers

1. Individual polling unit result page with chained LGA, ward, and polling unit selectors.
2. Summed LGA result page that computes totals from polling unit results only.
3. New polling unit submission page that stores scores for all parties in one action.

## Folder analysis summary

- The workspace originally contained only one file: `bincom_test.sql`.
- The dump includes Delta State election data across `states`, `lga`, `ward`, `polling_unit`, `party`, `announced_pu_results`, and `announced_lga_results`.
- There are 25 Delta LGAs, 263 wards, 272 polling unit rows, and 150 announced polling unit result rows.
- Only a subset of polling units have announced results, and many polling unit rows are placeholders with empty identifiers.
- The party table uses `LABOUR`, while the result rows use `LABO`; the app normalizes that mismatch.

## Architecture step-by-step

1. Create Django models that mirror the supplied SQL schema.
2. Build a custom management command to parse MySQL-style `INSERT INTO` blocks from `bincom_test.sql`.
3. Import the SQL dump into SQLite so the app can run locally and on Render without needing MySQL.
4. Filter selector options to usable Delta State polling units and announced-result records.
5. Render the three required pages with responsive templates and chained combo-box behavior.
6. Prepare deployment with `gunicorn`, WhiteNoise, static collection, and a `render.yaml`.

## Run locally

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-analytics.txt
python manage.py migrate
python manage.py seed_demo_data
python manage.py runserver
```

Open `http://127.0.0.1:8000/`.

If you only need the lean web runtime, install:

```bash
pip install -r requirements.txt
```

## Analytics stack

If you want the environment ready for data analysis, notebooks, and lightweight ML experiments as well, install:

```bash
pip install -r requirements-analytics.txt
```

This keeps Render deployment lean by leaving the web app runtime in `requirements.txt` and the heavier analysis stack in `requirements-analytics.txt`.

To export quick analysis tables from the imported SQLite database:

```bash
python scripts/analyze_election_data.py
```

This writes CSV summaries into `analysis_outputs/`.

To build the LGA-level modeling dataset used by the ML submission:

```bash
python scripts/build_lga_modeling_dataset.py
```

To regenerate the Bincom-specific submission notebook from source:

```bash
python scripts/render_ml_submission_notebook.py
```

To execute the completed notebook from top to bottom:

```bash
jupyter nbconvert --to notebook --execute --inplace ML_Submission_Template.ipynb
```

The notebook and helper scripts implement:

1. EDA across all 25 Delta LGAs.
2. Missing-result coverage analysis.
3. LGA-level feature engineering for the 8 result-backed LGAs.
4. StandardScaler vs MinMaxScaler comparison.
5. Hierarchical clustering with K-means elbow diagnostics.
6. Cluster strategy briefs exported into `analysis_outputs/cluster_profiles.csv`.

To generate a data-quality report for the imported dataset:

```bash
python manage.py audit_bincom_data
```

This writes `analysis_outputs/data_quality_report.json` and highlights the known inconsistencies in the source dump.

## Data-quality notes

- The source dump contains `170` placeholder polling-unit rows with blank labels or zero IDs.
- `polling_unit_id` is reused across many LGAs, so the app treats `polling_unit.uniqueid` as the true identity field.
- One known polling unit has an LGA/ward mismatch: `polling_unit.uniqueid=16` points to `lga_id=7`, but its ward belongs to `lga_id=6`.
- The party table uses `LABOUR`, while result tables use `LABO`; the app normalizes both to one canonical code.
- `announced_lga_results.lga_name` stores numeric LGA IDs as strings, not human-readable names.
- Announced LGA totals are shown only as a cross-check because polling-unit coverage is incomplete.

## Render deployment

`render.yaml` is included for Render deployment. The web service is configured for a demo-style SQLite deployment where the SQL dump is seeded only when the database is empty.

1. Push the repository to GitHub.
2. In Render, create a new Web Service from the GitHub repository.
3. Let Render read `render.yaml`, or copy the same build and start commands manually.
4. Confirm these environment variables are present:
   `DEBUG=False`
   `SECRET_KEY=<generated by Render>`
   `ALLOWED_HOSTS=.onrender.com`
   `CSRF_TRUSTED_ORIGINS=https://bincom-election-explorer.onrender.com`
5. Deploy the service and wait for the initial build to finish.
6. Smoke-test the deployed routes:
   `/`
   `/polling-units/`
   `/lga-results/`
   `/polling-units/new/`

## Deployment caveat

This Render setup is suitable for assessment/demo hosting. It does **not** provide durable database persistence for new polling-unit submissions across redeploys or infrastructure resets because it uses SQLite instead of a managed database.
